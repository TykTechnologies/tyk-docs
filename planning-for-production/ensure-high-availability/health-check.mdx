| Edition | Deployment Type |
| :------------- | :---------------------- |
| Community & Enterprise | Self-Managed, Hybrid |

Health checks verify the status of Tyk components (Gateway, Dashboard, Pump) and determine whether they are ready to process traffic or are simply running in an unresponsive or idle state.

### Use cases

- **Load Balancer Configuration**: Route traffic only to healthy instances.
- **Kubernetes Probes**: Automatically restart unhealthy pods or remove them from service.
- **Monitoring and Alerting**: Track the status of Tyk components and their dependencies (like Redis, RPC).

## Health Check Endpoints

Tyk provides two distinct HTTP endpoints for health checks:

- **Liveness Probe (`/hello` or `/health`)**: Indicates if the Tyk process is running and its web server is responsive. It returns HTTP 200 OK as long as the process is alive, regardless of dependency status like Redis.
- **Readiness Probe (`/ready`)**: <Badge color="orange">Gateway Only</Badge> Indicates if the Gateway is fully initialized and ready to process traffic. It returns a non-200 status code (e.g., 503 Service Unavailable) if critical dependencies are not available. The Gateway is considered "ready" only when its dependencies are healthy and the API definitions are loaded.

<Note>
Health check is implemented as per the [Health Check Response Format for HTTP APIs](https://inadarei.github.io/rfc-healthcheck/) RFC.
</Note>

### Available Endpoints

| Component | Liveness Endpoint | Readiness Endpoint | Configuration Key | Introduced In |
|-----------|-------------------|--------------------|-------------------|---------------|
| **Gateway** | `/hello` | `/ready` | [health_check_endpoint_name](/tyk-oss-gateway/configuration#health_check_endpoint_name) / [readiness_check_endpoint_name](/tyk-oss-gateway/configuration#readiness_check_endpoint_name) | Liveness: v2.7.5<br/>Readiness: v5.8.3+ |
| **Dashboard** | `/health` | N/A | [`health_check_endpoint_name`](/tyk-dashboard/configuration#health_check_endpoint_name) | v3.0.0 |
| **Pump** | `/health` | N/A | [`health_check_endpoint_name`](/tyk-pump/tyk-pump-configuration/tyk-pump-environment-variables#health_check_endpoint_name) | Liveness: v2.9.4 |

The default endpoint paths can be changed via the `health_check_endpoint_name` config key (all components) and `readiness_check_endpoint_name` (Gateway only). For example:

```yaml
health_check_endpoint_name: "status"
```

This changes the liveness endpoint to `/status`. For full details, refer to the [Gateway](/tyk-oss-gateway/configuration), [Dashboard](/tyk-dashboard/configuration), and [Pump](/tyk-pump/tyk-pump-configuration/tyk-pump-environment-variables) configuration references.

### Response Format

#### Gateway

The Gateway's `/hello` and `/ready` endpoints return a structured JSON response. Health status is cached and refreshed every 10 seconds. Both endpoints reflect the most recent snapshot:

```json
{
  "status": "pass",
  "version": "v5.8.10",
  "description": "Tyk GW",
  "details": {
    "redis": { "status": "pass", "componentType": "datastore", "time": "..." }
  }
}
```

The `status` field can have the following values:

| Status | HTTP Code | Description |
|--------|-----------|-------------|
| **pass** | 200 | All dependencies available. No impact on traffic. |
| **warn** | 200 | One dependency is down but Gateway continues processing. Medium impact (no quotas, no analytics, no RPC connection to [MDCB](/api-management/mdcb)). |
| **fail** | 200 (`/hello`) / 503 (`/ready`) | Critical dependencies (Redis and/or the Tyk Dashboard) unavailable. High impact (no configuration changes, no quotas, no analytics). |

#### Dashboard, Pump

Both return a simple live response with no caching:

```json
{"status": "ok"}
```

HTTP 200 is returned when the service is running.

## Deployment Configuration

### Non-Kubernetes

Point your load balancer or external monitoring system at `/hello` to check Gateway health. The endpoint names can also be set via environment variables instead of the config file:

```bash
TYK_GW_HEALTHCHECKENDPOINTNAME=status
TYK_GW_READINESSCHECKENDPOINTNAME=is-ready
```

### Kubernetes (Helm)

When deploying Tyk Gateway on Kubernetes, use `/hello` for the liveness probe and `/ready` for the readiness probe. Configure them via `gateway.livenessProbe` and `gateway.readinessProbe` in your Helm `values.yaml`:

```yaml
gateway:
  livenessProbe:
    httpGet:
      path: /hello
      port: 8080
    initialDelaySeconds: 10
    periodSeconds: 10
    timeoutSeconds: 5
    failureThreshold: 3
  readinessProbe:
    httpGet:
      path: /ready
      port: 8080
    initialDelaySeconds: 15
    periodSeconds: 10
    timeoutSeconds: 5
    failureThreshold: 3
```

<Note>
Adjust `initialDelaySeconds` based on the number of APIs your Gateway loads and available pod resources. If using the Tyk Operator, probe configuration is still done at the Helm level; the Operator manages API definitions and policies, not the Gateway deployment itself.
</Note>

The following examples show how the Health check endpoint returns for the Gateway. For a description of the `status` values and HTTP codes, see [Response Format](#response-format).

The following is returned for a `pass` status level for the Open Source Gateway:

```http

{
  "description": "Tyk GW",
```
