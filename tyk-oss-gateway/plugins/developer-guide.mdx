
# Tyk Gateway Plugin Developer Guide

This guide provides detailed technical information for developers building custom plugins for the Tyk Gateway. It covers the middleware execution order, data availability, and the various hooks available for custom plugins.

## Middleware Execution Order

The Tyk Gateway processes requests through a chain of middleware. The order of execution is critical for plugin developers to understand, as it determines when a plugin can interact with a request. The following is the precise order of middleware execution:

1.  **Recovery**: Panic recovery middleware.
    *   **Data Available**: Raw request object.
2.  **ProxyLogger**: Logs incoming request details.
    *   **Data Available**: Raw request object, session object (if available).
3.  **CheckHost**: Verifies the `Host` header.
    *   **Data Available**: Raw request object, session object.
4.  **RateLimiter**: Enforces API key rate limits.
    *   **Data Available**: Request context, session object, metadata.
5.  **KeyExists**: Handles authentication and authorization.
    *   **Data Available**: Request context, session object, metadata.
6.  **ScopeHandler**: Checks OAuth scopes.
    *   **Data Available**: Request context, session object, metadata.
7.  **ContentSniffer**: Inspects request content.
    *   **Data Available**: Request context, session object, metadata, request body.
8.  **APILimiter**: Enforces global API rate limits.
    *   **Data Available**: Request context, session object, metadata.
9.  **FinalMiddleware**: Proxies the request to the upstream service.
    *   **Data Available**: Request context, session object, metadata, response object (after upstream).

## Data Availability

In the context of a custom plugin, several data objects are available. Understanding what data is available at each stage of the middleware chain is crucial for plugin development.

*   **Session Object**: The session object contains information about the API key, including rate limits, access rights, and metadata. It is available after the `KeyExists` middleware has run.
*   **Request Context**: The request context contains information about the incoming request, including headers, query parameters, and the request body. It is available from the beginning of the middleware chain.
*   **Metadata**: The metadata object is a key-value store that can be used to pass data between middleware. It is available from the beginning of themiddleware chain and can be modified by any middleware.

## Custom Plugin Hooks

The Tyk Gateway provides several hooks that allow you to execute custom code at different points in the request lifecycle.

*   **Pre-hooks**: These hooks are executed before the main middleware chain. They are useful for modifying the incoming request before it is processed by the gateway.
*   **Post-hooks**: These hooks are executed after the main middleware chain, but before the response is sent to the client. They are useful for modifying the response from the upstream service.
*   **Custom Middleware**: You can also create your own custom middleware and insert it into the middleware chain at any point. This gives you the ultimate flexibility to customize the request processing logic.

By understanding the middleware execution order, data availability, and the various plugin hooks, you can build powerful custom plugins that extend the functionality of the Tyk Gateway.
