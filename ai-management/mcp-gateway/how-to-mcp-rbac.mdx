After [securing your MCP proxy](/ai-management/mcp-gateway/how-to-proxy-remote-mcp), the next step is controlling what each consumer can do. Role-based access control (RBAC) in Tyk MCP lets you give different agents different views of the same proxy — without creating separate proxy definitions. A policy bound to a key determines which tools that agent can invoke.

This guide creates two roles on the Mock MCP Server:

- **Reader** — can only call `get_users`, `get_posts`, `get_products`, and `get_analytics`
- **Admin** — can call all 15 tools

You'll create a policy for each role, issue role-specific keys, then use MCP Inspector to verify that each key sees exactly the tools it is permitted to access.

---

## How it works

Tyk policies control consumer access at two levels relevant to this guide:

**Primitive access** — restricts which specific tools a consumer can invoke. When a key's policy includes an allowed list for tools, Tyk enforces it on both `tools/call` (blocking disallowed tool invocations) and `tools/list` (filtering the response so the agent only sees tools it can use). The upstream server is never reached for blocked calls.

**Proxy access** — determines which MCP proxies the key can reach at all.

Both keys in this guide point at the same proxy URL. The difference in behaviour comes entirely from the policies applied to each key.

For the complete policy reference, see [MCP policies](/ai-management/mcp-gateway/policies).

## Before you begin

- The Mock MCP Server running on `http://localhost:7878` — set up in the [quickstart](/ai-management/mcp-gateway/quickstart)
- An MCP proxy named **Mock MCP Server** with authentication enabled — see [How to secure an MCP proxy](/ai-management/mcp-gateway/how-to-proxy-remote-mcp)
- Node.js 18 or later (to run MCP Inspector)
- A Dashboard user account with policy management permissions

## Step 1: Create the Reader policy

In the Tyk Dashboard sidebar, click **Policies**, then click **Add Policy**.

On the **Access Rights** tab, find **Mock MCP Server** in the API list and click it to add it.

![Select Mock MCP Server from the API list](/img/ai-management/tyk-how-to-rbac-select-api.png)

The proxy expands to show its configuration options.

### Add the tool allowlist

Scroll to the **Primitive based access** section within the Mock MCP Server panel. Add each permitted tool:

1. Click **Add**, enter `get_users`, set **Type** to **Tool**, and set the status to **Allowed**. Click **Add**.
2. Repeat for `get_posts`.
3. Repeat for `get_products`.
4. Repeat for `get_analytics`.

![Primitive based access configuration](/img/ai-management/mcp-how-to-rbac-primitive.png)

Once you add any tool with **Allowed** status, Tyk treats the list as an explicit allowlist for that consumer — any tool not in the list is blocked. The remaining 11 tools on the Mock MCP Server, including `create_user`, `delete_user`, and `process_order`, will be inaccessible to keys on this policy.

On the **Configurations** tab:

- **Policy Name**: `Reader`
- **Policy State**: **Active**

Click **Create Policy**.

![Reader policy configuration](/img/ai-management/mcp-how-to-rbac-reader.png)

---

## Step 2: Create the Admin policy

The Admin policy grants unrestricted tool access. No primitive access configuration is needed — omitting the **Primitive based access** entries means all tools are accessible.

Click **Add Policy**.

On the **Access Rights** tab, add **Mock MCP Server**.

On the **Configurations** tab:

- **Policy Name**: `Admin`
- **Policy State**: **Active**

Click **Create Policy**.

---

## Step 3: Issue role-specific keys

In the Dashboard sidebar, click **Keys**, then **Add Key**.

Under **Access rights**, click **Apply Policy** and select **Reader**.

![Apply Reader policy to key](/img/ai-management/mcp-how-to-rbac-keys.png) Click the **Configurations** tab and set an **Alias** such as `reader-agent` to identify this key in analytics.

Click **Create Key** and copy the key.

![Set alias and create key](/img/ai-management/mcp-how-to-rbac-set-alias.png)

Repeat to issue a second key, selecting **Admin** as the policy. Set the alias to `admin-agent`.

You now have two keys with different tool access, both pointing at the same proxy endpoint.

---

## Step 4: Verify in MCP Inspector

Start MCP Inspector:

```bash
npx @modelcontextprotocol/inspector
```

Open the URL printed in your terminal.

### Test the Reader role

In the Inspector UI:

1. Set **Transport Type** to `Streamable HTTP`
2. Set **URL** to your MCP endpoint (find it under **MCP Proxy URL** in the proxy designer, then append `/mcp`)
3. Add a header: `Authorization` = `Bearer {reader-api-key}`
4. Click **Connect**

Click the **Tools** tab. You will see exactly four tools: `get_users`, `get_posts`, `get_products`, and `get_analytics`. The other 11 tools are absent — Tyk has filtered the `tools/list` response based on the Reader policy's allowed list.

Select `get_users` and click **Run**. It succeeds. The Reader can call any tool on its allowed list.

### Test the Admin role

Click **Disconnect**. Replace the key in the `Authorization` header with your Admin key and click **Connect** again.

Click the **Tools** tab. All 15 Mock MCP Server tools appear. The Admin policy applies no tool restrictions.

Both keys connect to the same proxy at the same URL. The difference in tool availability is driven entirely by the policy.
