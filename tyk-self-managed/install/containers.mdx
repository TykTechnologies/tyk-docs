**Goal:** Make it crystal clear **what is supported, what isn't, and what's safe** when moving between Docker, containerd/CRI-O, and Podman.

## 1) Where Tyk fits in your stack

Tyk is made of deployable components that run **as containers** or **as native services**:

- **Data plane:** **Tyk Gateway** (ingresses API traffic)
- **Control plane (Self-Managed):** **Tyk Dashboard/Manager** (+ **MDCB** for multi-cluster control)
- **Observability:** **Tyk Pump** (ships analytics to your sink)

You can run everything with **Docker/Podman** on a single host for demos, with **Docker Compose** for small dev setups, or with **Helm on Kubernetes** for production. (See: Tyk Compose quick starts and Helm charts.)

> **Takeaway:** Tyk doesn't require Docker specifically. It runs anywhere the **underlying container runtime supports OCI images** (containerd, CRI-O, Docker Engine).

---

## 2) The container runtime landscape (why Docker ≠ default in Kubernetes)

- Since **Kubernetes v1.24**, the **dockershim** compatibility layer was removed upstream. Managed K8s moved to **CRI-compliant runtimes** (mostly **containerd** or **CRI-O**). Your images still work because they're **OCI**.
- **Amazon EKS, Google GKE, Azure AKS** all use **containerd** by default now. **OpenShift** uses **CRI-O**.

> **What this means for you:** keep building "Docker images" (they're really **OCI** images) and run them on EKS/GKE/AKS/OpenShift with no change.

---

## 3) Local vs. containerized vs. Kubernetes

### Local (single host)

- **Fastest path** to try Tyk: Docker or Podman on your laptop/VM. Our quick starts and demo repos use **Docker Compose** by default.
- Swap Docker with **Podman** on Linux (RHEL 8+, Fedora, etc.)—same images, same ports, similar CLI. Details below.

### Containerized (Compose)

- Use `docker compose up` (or `podman compose up`) to bring up **Gateway + Redis** and optionally **Dashboard + Pump** for dev demos. Same compose.yaml works for both, see Podman notes.

### Kubernetes (Helm)

- For production, install with our **Helm charts**; this works across EKS/GKE/AKS and OpenShift (CRI-O).
- **MDCB** adds multi-cluster control planes and remote data planes.

## 4) Building images (Docker, Podman, Buildah)

- **Build tools:** `docker build` or `podman build` both emit **OCI-compliant images** that run on containerd/CRI-O. Use your usual CI runners.
- **Registries:** ECR/ACR/Artifact Registry support OCI (incl. OCI 1.1 referrers for SBOMs/signatures).

**Examples**

```bash
# Build with Docker
docker build -t your-registry/tyk-gateway:TAG .

# Build with Podman (RHEL 8+)
podman build -t your-registry/tyk-gateway:TAG .
```

## 5) Running with Docker Compose (dev)

Minimal example for **Gateway + Redis**:

```yaml
# compose.yaml
services:
  redis:
    image: redis:7-alpine
  gateway:
    image: tykio/tyk-gateway:latest
    ports: ["8080:8080"]
    volumes:
      - ./tyk.standalone.conf:/opt/tyk-gateway/tyk.conf:ro
    depends_on: [redis]
```

- **Docker:** `docker compose up -d`
- **Podman:** `podman compose up -d` (Podman supports running Compose files)

Official images: `tykio/tyk-gateway`, `tykio/tyk-dashboard` on Docker Hub.

## 6) Kubernetes (prod): runtimes & verification

- EKS/GKE/AKS use **containerd**; OpenShift uses **CRI-O**. (Table below.)

```bash
kubectl get node -o jsonpath='{range .items[*]}{.metadata.name}{"  "}{.status.nodeInfo.containerRuntimeVersion}{"\n"}{end}'
```

## 7) Podman on RHEL 8: how to run Tyk safely

RHEL 8 ships **Podman** (daemonless, rootless-friendly). It replaces Docker CLI with Podman/Buildah/Skopeo tooling.

### Common run pattern (rootless)

```bash
# SELinux relabel with :Z so the container can write to the host path
podman run -d --name tyk-gateway \
  -p 8080:8080 \
  -v $PWD/tyk.standalone.conf:/opt/tyk-gateway/tyk.conf:Z \
  tykio/tyk-gateway:latest
```

SELinux requires `:Z` / `:z` volume labels for bind mounts so the container can access files.

### Compose on Podman

Use **`podman compose up`**, or (if you must) run Docker Compose by exposing Podman's Docker-compatible service/socket.

### Moving between Podman ↔ Kubernetes

- **Generate** Kubernetes YAML from local containers: `podman kube generate …`
- **Play** Kubernetes YAML locally: `podman kube play …`
Great for prototyping Tyk locally, then pushing to a real cluster.

## 8) "What it is / isn't" and "what's safe / not"

### What it is

- **OCI everywhere.** If you build a standard image, it runs on Docker, containerd, or CRI-O. Kubernetes no longer needs Docker.
- **Podman compatibility.** Podman's CLI is Docker-like; your images and Compose files generally "just work."

### What it isn't

- **Podman is not a K8s runtime.** Kubelet talks CRI to containerd/CRI-O; Podman doesn't implement CRI. (On OpenShift, the runtime is **CRI-O**.)
- **"Docker required"** is outdated for K8s. Managed clusters won't run Docker Engine on nodes by default.

### Safe defaults (do this)

- **Run as non-root** (works on OpenShift with arbitrary UID).
- **Use SELinux labels** on RHEL/Fedora bind mounts (`:Z` or `:z`).
- **Pin images by digest** in production and keep SBOM/signatures in OCI-aware registries (OCI 1.1 referrers).
- **Use Helm** for production installs/upgrades on K8s.

### Risky/avoid (not safe by default)

- Depending on **/var/run/docker.sock** inside containers (won't exist on containerd/CRI-O).
- Assuming static UIDs or writing to read-only paths on OpenShift (use writable volumes, group-writable paths). (See SCC/arbitrary-UID guidance.)
- Skipping SELinux relabeling on RHEL/Fedora when bind mounting host paths.

## 9) Quick reference: default runtimes by major providers

| Provider | Default runtime (Linux nodes) | Notes / source |
| :--- | :--- | :--- |
| **AWS EKS** | **containerd** | AWS moved customers to containerd after dockershim removal; EKS AMIs (Linux & Windows) include containerd. |
| **Google GKE** | **containerd** | containerd default since GKE 1.19 (Linux); Docker images deprecated by 1.24. |
| **Azure AKS** | **containerd** | Linux uses containerd (≥1.19). Windows uses containerd (only option ≥1.23). |
| **Red Hat OpenShift** | **CRI-O** | OpenShift (incl. ROSA) uses CRI-O with runc/crun. |
| **DigitalOcean Kubernetes** | **containerd** | Switched to containerd in 1.20 (2021). |
| **IBM Cloud Kubernetes Service** | **containerd** | Docs show kubelet's containerRuntimeEndpoint points to containerd. |
| **Rancher (RKE2/K3s)** | **containerd** | Defaults to containerd |
| **VMware Tanzu** | **containerd** | Defaults to containerd |

> If you're on another provider, use the `kubectl ... jsonpath` one-liner in §6 to confirm the runtime on your nodes.

## 11) FAQ

**Q: Do we have to rebuild images to move from Docker → containerd/CRI-O?**

**A:** No. Docker-built images are **OCI**; they run unchanged on containerd/CRI-O.

**Q: Does Podman change how our app runs?**

**A:** Functionally no—images are the same. On RHEL, pay attention to **rootless defaults** and **SELinux volume labels** (`:Z` / `:z`).

**Q: Can we keep using Docker Compose on Podman?**

**A:** Yes. Use `podman compose …` or point Docker Compose at Podman's service/socket; see Podman docs.

**Q: How do we know what runtime our cluster uses?**

**A:** Run the `kubectl … jsonpath` one-liner from §6.

### Appendix: Why images are portable

The **Open Container Initiative (OCI)** standardizes image formats and distribution so that tools like Docker, Podman, containerd, CRI-O, and registries (ECR/ACR/etc.) interoperate. That's why you can build with Docker or Podman and run on any modern Kubernetes.
