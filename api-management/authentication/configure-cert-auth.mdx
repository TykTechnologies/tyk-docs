## Prerequisites

- An active Tyk installation (Gateway and Dashboard).
- An existing API configured in Tyk. For this tutorial, we'll assume it's an API proxying `http://httpbin.org` and is currently set to use "Open (No Auth)".
- `openssl` installed on your local machine to generate the certificate.

## Instructions

### Step 1: Create a Client Certificate (Optional)

If you don't already have a client certificate, you can generate a self-signed one for testing purposes. This involves creating a private key and then a public certificate.

1.  **Generate a Private Key:**

    ```bash
    openssl genrsa -out client.key 2048
    ```

2.  **Generate a Certificate Signing Request (CSR):**

    ```bash
    openssl req -new -key client.key -out client.csr
    ```
    You will be prompted to enter information for the certificate, such as country, organization, etc. You can fill these in or leave them as default.

3.  **Generate the Self-Signed Certificate:**

    ```bash
    openssl x509 -req -days 365 -in client.csr -signkey client.key -out client.crt
    ```

    This creates a certificate (`client.crt`) that is valid for 365 days. You now have `client.key` (your private key) and `client.crt` (your public certificate).

### Step 2: Register the Certificate in Tyk Dashboard

Now, you need to tell Tyk about this certificate so it can trust clients who present it.

1.  Navigate to **TLS/SSL Certificates** in your Tyk Dashboard.
2.  Click the **Add Certificate** button.
3.  Select the `client.crt` file you generated in Step 1 and upload it.
4.  Click **Save Certificate**. The certificate will be added to the list with a unique ID.

### Step 3: Configure Your API to Use Certificate Authentication

Next, let's switch your API's authentication mode.

1.  Go to **APIs** and select the API you want to configure.
2.  Under the **Authentication** section, select **Certificate Based Authentication** from the dropdown menu.
3.  Click **Save** to apply the changes.

Your API is now protected and will require clients to present a valid, registered certificate.

### Step 4: Create a Key and Associate it with the Certificate

Even with certificate authentication, Tyk uses an API Key to apply policies and manage access rights. We need to create a key and link it to the certificate we registered.

1.  Navigate to **System Management > Keys**.
2.  Click **Add Key**.
3.  In the **API Access** section, select your API from the dropdown list to grant this key access.
4.  Scroll down to the **Advanced Options** section.
5.  In the **Client Certificate** dropdown, you will see the certificate you added in Step 2. Select it.
6.  Click **Create Key**. Copy the generated key, as you will need it for testing.

This key is now tied to the specific client certificate. It will only grant access if the request includes both a valid API key and the correct client certificate.

---

### Step 5: Test the Secured API

Let's make sure everything is working. We'll use `curl` to test. You will need your Gateway domain, API listen path, the API key you just created, and the client certificate files (`client.crt` and `client.key`).

#### 1. Successful Request (with Certificate and Key)

Replace `YOUR_GATEWAY_URL`, `YOUR_API_LISTEN_PATH`, and `YOUR_API_KEY` with your actual values.

```bash
curl --cert client.crt --key client.key -H "Authorization: YOUR_API_KEY" YOUR_GATEWAY_URL/YOUR_API_LISTEN_PATH/
```

If successful, you should receive a `200 OK` response from `httpbin.org`.

#### 2. Failed Request (without Certificate)

Now, let's try making a request with only the API key.

```bash
curl -H "Authorization: YOUR_API_KEY" YOUR_GATEWAY_URL/YOUR_API_LISTEN_PATH/
```

You should receive a `401 Unauthorized` error with the message `"error": "Client certificate not found"`.

### Step 2: Create a key

**Registering Certificate Authentication User Credentials**

The *dynamic allow list* comprises session state objects in the Gateway's temporal storage (typically Redis) that reference the client certificates that should be accepted.

1. First you must [register](/api-management/certificates#tyk-certificate-store-api) the client certificate with the Tyk Certificate Store and note the certificate ID that is assigned.

2. Next you *create a key*, providing the certificate ID in the `certificate` field of the session object payload.
    - Tyk Gateway API: `POST /tyk/keys/create`
    - Tyk Dashboard API: `POST /api/keys/create`

3. Tyk will create a session object with the Redis key containing the certificate ID, which forms part of the dynamic allow list.
    - the Redis key is formed from a combination of the Organization ID and Certificate ID
    - deleting this object (key) will remove the certificate from the allow list, restricting access to any client presenting that certificate

From the Tyk Dashboard UI, if you [create a key](/getting-started/using-tyk-dashboard#api-security) that grants access to an API secured with a dynamic allow list, the **Authentication** tab will be displayed, where you can select the client certificate from the Tyk Certificate Store.
