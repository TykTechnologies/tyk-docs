## Overview

When you configure an API on Tyk, the Gateway will proxy all requests received at the listen path that you have defined through to the upstream (target) URL configured in the API definition. Responses from the upstream are likewise proxied on to the originating client. Requests and responses are processed through a powerful chain of middleware that perform security and processing functions.

Within that chain are a highly configurable set of optional middleware that can, on a per-endpint basis:
- apply processing to API requests before they are proxied to the upstream service
- apply customization to the API response prior to it being proxied back to the client

Tyk also supports a powerful custom plugin feature that enables you to add custom processing at different stages in the processing chains. For more details on custom plugins please see the [dedicated guide](/api-management/plugins/overview#).

### Custom Plugin Injection Points (Hooks)

Custom plugins (Go, Python, gRPC) can be injected at the following points in the middleware chain:

*   **`Pre`:** Executes before any built-in middleware in the `Pre` stage.
*   **`AuthCheck`:** Executes as a custom authentication mechanism.
*   **`PostKeyAuth`:** Executes after successful authentication but before other `PostKeyAuth` middleware.
*   **`Post`:** Executes at the end of the `Post` stage, before the request is proxied.
*   **`Response`:** Executes on the response from the upstream service.
