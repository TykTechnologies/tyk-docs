## Overview

The v1 JSVM upgrade ships two related changes to Tyk Gateway:

1. **The engine backing the `javascript` driver swaps from [otto](https://github.com/robertkrimen/otto) to [goja](https://github.com/dop251/goja)** â€” same API definition contract, faster runtime, ES5.1+ compliance.
2. **A new `code` field on each middleware entry** lets you embed the plugin source directly in the API definition instead of shipping a bundle.

Together this means you can ship a JS plugin to the Gateway without operating a bundle server, hosting a `.zip`, or touching the Dashboard. The plugin's source travels with the API definition.

<Note>
**v1 scope:** Gateway engine upgrade only. The Dashboard does not yet provide a UI for editing the `code` field â€” you ship inline plugins by writing the OAS document directly (file mount or the Gateway control API).
</Note>

## When to use this

Use inline `code` when:

- You want a single, self-contained OAS document that includes its custom logic
- You don't want to operate a bundle server or expose plugin artifacts over HTTP
- The plugin is small enough that embedding the source in the OAS document is reasonable

Use a [bundled plugin](/api-management/plugins/rich-plugins/) instead when:

- You need to share one plugin across many APIs
- The compiled code is large
- You want plugin artifacts versioned and distributed independently of API definitions

## How it works

1. You author the plugin in TypeScript/JavaScript (see [tyk-plugin-starter](https://github.com/TykTechnologies/tyk-plugin-starter)).
2. You compile it to a single `plugin.js` file.
3. You base64-encode `plugin.js` and put the encoded string in the `code` field of an entry under `x-tyk-api-gateway.middleware.global.<hook>Plugins`.
4. You set `x-tyk-api-gateway.middleware.global.pluginConfig.driver` to `javascript`.
5. You POST or PUT the OAS document to the Gateway's OAS endpoint and reload (`/tyk/reload/group`).

On load, the Gateway decodes the base64, compiles the source under a unique synthetic path (`__tyk_inline__/<api_id>/<counter>/<name>`), and binds the exported handler to the named hook. Each inline entry gets its own isolated namespace, so multiple plugins that all declare `var handler = ...` don't collide.

### Field reference

Each entry in `prePlugins` / `postAuthenticationPlugins` / `postPlugins` / `responsePlugins`:

| Field | Type | Notes |
| --- | --- | --- |
| `enabled` | bool | Required. Set to `false` to disable without removing the entry. |
| `functionName` | string | The handler identifier (e.g. `handler`). The Gateway invokes `<functionName>.DoProcessRequest(...)` at runtime. |
| `code` | string | Base64-encoded JS source. When set, `path` is ignored and no bundle is fetched for this entry. |
| `path` | string | Path to the JS file on disk or inside a bundle. Mutually exclusive with `code`. |
| `requireSession` | bool | Pre-load the session before invoking the hook. |
| `rawBodyOnly` | bool | Skip request-body decoding (useful for large or non-text payloads). |

`pluginConfig.driver` must be `javascript`. On this branch that value routes to the goja engine; `otto` continues to select the legacy engine for backward compatibility.

### Supported hook keys

| OAS key | Classic equivalent |
| --- | --- |
| `prePlugins` | `custom_middleware.pre` |
| `postAuthenticationPlugins` | `custom_middleware.post_key_auth` |
| `postPlugins` | `custom_middleware.post` |
| `responsePlugins` | `custom_middleware.response` |

All accept inline `code`. (Inline `auth_check` is also supported on the classic API definition shape; OAS exposes auth-check plugins via `x-tyk-api-gateway.server.authentication.custom`.)

## Authoring workflow

1. Scaffold from [tyk-plugin-starter](https://github.com/TykTechnologies/tyk-plugin-starter): `npx degit TykTechnologies/tyk-plugin-starter my-plugin`
2. Edit `src/plugin.ts` â€” request/session/config types come from `@tyk-technologies/tyk-plugin-types`.
3. `npm test` â€” the harness simulates the JSVM so you can unit-test without a Gateway.
4. `npm run build` â€” emits `dist/plugin.js`.
5. Base64-encode the output and embed it into the OAS document's `code` field:

```bash
base64 < dist/plugin.js | tr -d '\n'
```

## Updating a plugin

Inline plugins live in the OAS document, so updates are a normal OAS update:

```bash
curl -X PUT \
  -H "x-tyk-authorization: $TYK_SECRET" \
  -H "Content-Type: application/yaml" \
  --data-binary @api.yaml \
  http://gateway:8080/tyk/apis/oas/header-rename
curl -H "x-tyk-authorization: $TYK_SECRET" http://gateway:8080/tyk/reload/group
```

The Gateway re-decodes and re-compiles the `code` field on hot reload. There is no on-disk plugin cache to invalidate.

## Combining inline and file-mounted plugins

Within a single hook list you can mix inline entries (`code` set) and file-mounted entries (`path` set, resolved relative to the gateway's working directory or given as an absolute path). The loader picks the inline branch when `code` is non-empty and the file branch otherwise. This is useful when most of your logic lives on disk but you want one small per-API tweak embedded in the OAS document.

**Inline `code` and bundles do not combine.** When `pluginConfig.bundle.enabled` is true and `pluginConfig.bundle.path` is set, the gateway fetches the bundle and the bundle's manifest replaces the API's `custom_middleware` section in its entirety (`gateway/coprocess_bundle.go` `AddToSpec`). Any inline `code` you set on the OAS document is overwritten and ignored. If you want a bundle, ship all your middleware in the bundle; if you want inline `code`, leave the bundle field unset.

`pluginConfig.bundle.path` (the classic field is `custom_middleware_bundle`) is a single string but accepts a comma-separated list of bundle names â€” `"auth.zip,logging.zip"` fetches both and merges their manifests in declaration order. This is a property of the bundle field, not of inline `code`; the same "bundle wins over inline" rule applies whether one or several bundles are listed.

## Limitations (v1)

- **No Dashboard UI**: inline plugins are authored as OAS documents; the Dashboard does not yet provide an editor for `code`.
- **No drift detection between source and compiled output**: if you forget to rebuild before base64-encoding, the Gateway happily runs the stale code. Treat the build step as part of your release process.
- **Source is visible to anyone with read access to the OAS document** â€” don't embed secrets.
- **Per-API only**: there is no mechanism to share one `code` block across multiple APIs. For that, use a bundle (or comma-separated multi-bundle composition).
- **Virtual endpoints follow the same driver flag**: a virtual endpoint runs on goja only when its API sets `pluginConfig.driver` (classic: `custom_middleware.driver`) to `javascript` â€” the same flag that selects goja for custom middleware. With no driver set, virtual endpoints default to the legacy otto engine, so you must opt in explicitly to move them onto goja.
