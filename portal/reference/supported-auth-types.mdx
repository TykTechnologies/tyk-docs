When you add APIs to an API Product, the Developer Portal enforces that all APIs in the product share a compatible authentication method. This page summarises which authentication types the Portal supports, what consumers and admins can do with each, and which types can be combined.

<Note>
This page covers authentication as managed by the Developer Portal. For Tyk Gateway-level authentication configuration, see [API Authentication](/api-management/authentication).
</Note>

## Credential Ownership: Self-Serve vs Managed

Before reviewing the table, it helps to understand the two credential ownership modes the Portal uses:

**Self-Serve:** The API Consumer creates, rotates, and revokes their own credentials from the Live Portal without Admin involvement.

**Managed:** The Portal Admin imports or pre-configures credentials. Consumers have view-only access, or can revoke but not create or rotate credentials.

The **Mode** column in the table below indicates which model applies to each authentication type.

## Supported Authentication Types

| Auth Type | Supported | Mode | Consumer Actions | Admin Actions |
|---|---|---|---|---|
| **API Token / API Key** | Yes | Self-Serve | Request access, rotate credential, revoke credential | View, edit, and revoke credentials |
| **OAuth 2.0 (via DCR)** | Yes | Self-Serve | Request access (generates Client ID and Secret), rotate secret, revoke credential | Configure DCR template, view credentials, revoke credentials |
| **JWT** | Yes | Self-Serve | Request access, rotate credential, revoke credential | Configure DCR template, view credentials, revoke credentials |
| **mTLS (Self-Serve)** | Yes | Self-Serve | Upload client certificate during access request, revoke access | View credentials, revoke access |
| **mTLS (Managed)** | Yes | Managed | Request access using organization certificates, view credentials, revoke access | Pre-configure and manage certificates per organization in the Admin Portal |
| **Auth Token + mTLS** | Yes | Self-Serve or Managed | Request access; issued credential is automatically bound to the consumer's certificate | Enable certificate-token binding on the Product |
| **Basic Authentication** | Yes | Managed | View credential details (read-only); revoke credential | Create, edit, and assign credentials via Admin Portal or Portal API |
| **Custom Authentication** | Yes | Managed | View credential details (read-only); revoke credential | Create, edit, and assign credentials via Admin Portal or Portal API |
| **Custom Credentials (Feature)** | Yes | Managed | View credential details (read-only); revoke credential; for Tyk-managed custom credentials: rotate credential | Create, edit, and assign credentials via Admin Portal or Portal API |

**Unsupported authentication types:** HMAC, OpenID Connect (OIDC), and Keyless are not supported by the Portal.

## Combination Rules

All APIs in an API Product must share a compatible authentication method. When a consumer requests access, the Portal issues a single credential that must work across every API in the product.

The rules below determine which authentication types can coexist in the same product.

<AccordionGroup>

<Accordion title="Single-type Products">
Any supported authentication type can be used alone in a product. There are no restrictions on single-type configurations.
</Accordion>

<Accordion title="Multi-type Products (mTLS combinations)">
mTLS must be present in any product that combines authentication types. The only supported multi-type combinations are:

- mTLS + API Key
- mTLS + JWT

API Key and JWT cannot be combined with each other unless mTLS is also present.
</Accordion>

<Accordion title="Auth types that cannot be combined">
The following authentication types must be used alone and cannot appear alongside any other auth type in the same product:

- Basic Authentication (`basicAuth`)
- Custom Authentication middleware (`customAuth`)
</Accordion>

<Accordion title="JWT requires DCR">
In the Portal, JWT APIs must have DCR enabled. The Portal rejects JWT APIs that do not have a DCR template configured at checkout. At checkout, the Portal sets the API's JWT source to the IdP's JWKS URI if not already configured.
</Accordion>

<Accordion title="DCR cross-IdP configurations">
A product can technically reference DCR templates from multiple Identity Providers, but this configuration does not work as intended. At checkout, the consumer selects one IdP. Tokens from that IdP are rejected by any API in the product configured to expect tokens from a different IdP. Use a single DCR template per product.
</Accordion>

</AccordionGroup>
