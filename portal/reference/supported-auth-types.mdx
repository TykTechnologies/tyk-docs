## Self-Serve vs Managed

**Self-Serve:** The API Consumer creates, rotates, and revokes their own credentials from the Live Portal without Admin involvement.

**Managed:** The Portal Admin imports or pre-configures credentials. Consumers have view-only access, or can revoke but not create or rotate credentials.

## Supported Authentication Types

| Auth Type | Supported | Mode | Consumer Actions | Admin Actions |
|---|---|---|---|---|
| **API Token / API Key** | Yes | Self-Serve | Request access, rotate credential, revoke credential | View, edit, and revoke credentials |
| **OAuth 2.0 (via DCR)** | Yes | Self-Serve | Request access (generates Client ID and Secret), rotate secret, revoke credential | Configure DCR template, view credentials, revoke credentials |
| **JWT** | Yes | Self-Serve | Request access, rotate credential, revoke credential | Configure DCR template, view credentials, revoke credentials |
| **mTLS (Self-Serve)** | Yes | Self-Serve | Upload client certificate during access request, revoke access | View credentials, revoke access |
| **mTLS (Managed)** | Yes | Managed | Request access using organization certificates, view credentials, revoke access | Pre-configure and manage certificates per organization in the Admin Portal |
| **Auth Token + mTLS** | Yes | Self-Serve or Managed | Request access; issued credential is automatically bound to the consumer's certificate | Enable certificate-token binding on the Product |
| **Basic Authentication** | Yes | Managed | View credential details (read-only); revoke credential | Create, edit, and assign credentials via Admin Portal or Portal API |
| **Custom Authentication** | Yes | Managed | View credential details (read-only); revoke credential | Create, edit, and assign credentials via Admin Portal or Portal API |
| **Custom Credentials (Feature)** | Yes | Managed | View credential details (read-only); revoke credential; for Tyk-managed custom credentials: rotate credential | Create, edit, and assign credentials via Admin Portal or Portal API |

**Unsupported Authentication Types:** HMAC, OpenID Connect (OIDC), and Keyless are unsupported by the Portal.

## Combination Rules

All APIs in an API Product must use a compatible authentication method. The Portal enforces this at checkout: a credential issued for one auth type cannot authenticate requests to an API expecting a different type.

**Single-type Products:** Any supported auth type can be used alone in a Product.

**Multi-type Products:** mTLS must be present in any Product that combines auth types. The only supported multi-type combinations are:

- mTLS + API Key
- mTLS + JWT

API Key and JWT cannot be combined without mTLS.

**Auth types excluded from all multi-type combinations:**

- Basic Authentication (`basicAuth`)
- Custom Authentication middleware (`customAuth`)

These auth types must be used alone and cannot be combined with any other auth type.

**JWT requirement:** JWT APIs must have DCR enabled. The Portal rejects JWT APIs that do not have a DCR template configured at checkout.

**DCR cross-IdP configurations:** A Product can technically reference DCR templates from multiple Identity Providers, but this configuration does not work as intended. At checkout, the consumer selects one IdP. Tokens from that IdP are rejected by any API in the Product configured to expect tokens from a different IdP. Use a single DCR template per Product.
