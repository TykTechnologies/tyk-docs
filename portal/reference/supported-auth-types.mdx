## Supported Authentication Types

| Auth Type | Supported | Mode | Consumer Actions | Admin Actions | Notes |
|---|---|---|---|---|---|
| **API Key** | Yes | Self-Serve | Request access, rotate credential, revoke credential | View, edit, and revoke credentials | Internal name in Portal product settings: Auth Token |
| **OAuth 2.0 (via DCR)** | Yes | Self-Serve | Request access (generates Client ID and Secret), rotate secret, revoke credential | Configure DCR template, view credentials, revoke credentials | Supports Authorization Code, Client Credentials, and Password grant types. Requires a DCR-configured Identity Provider. The Tyk internal OAuth provider (`oauth`) is not supported. |
| **JWT** | Yes | Self-Serve | Request access, rotate credential, revoke credential | Configure DCR template, view credentials, revoke credentials | JWT APIs must be DCR-enabled in the Portal. At checkout, the Portal sets the API's JWT source to the IdP's JWKS URI if not already set. |
| **mTLS (Self-Serve)** | Yes | Self-Serve | Upload client certificate during access request, revoke access | View credentials, revoke access | Consumer uploads their own certificate. The Portal registers it with the Tyk Certificate Store and adds it to the API allowlist. |
| **mTLS (Managed)** | Yes | Managed | Request access using organization certificates, view credentials, revoke access | Pre-configure and manage certificates per organization in the Admin Portal | Introduced in v1.16. Admin centrally manages certificates; consumers do not upload certificates. For JWT with mTLS APIs, certificate binding is managed in the Provider Dashboard, not the Portal. |
| **Auth Token + mTLS** | Yes | Self-Serve or Managed | Request access; issued credential is automatically bound to the consumer's certificate | Enable certificate-token binding on the Product | Introduced in v1.17. Requires Tyk Gateway 5.12.0 or later. The binding mode follows the mTLS mode configured on the Product. See [TLS Certificates](/portal/certificates). |
| **Basic Authentication** | Yes | Managed | View credential details (read-only); revoke credential | Create, edit, and assign credentials via Admin Portal or Portal API | Explicitly rejected during self-serve checkout. Must be provisioned by an Admin. Cannot be combined with other auth types. |
| **Custom Authentication** | Yes | Managed | View credential details (read-only); revoke credential | Create, edit, and assign credentials via Admin Portal or Portal API | Explicitly rejected during self-serve checkout. Must be provisioned by an Admin. Cannot be combined with other auth types. |
| **Custom Credentials (Feature)** | Yes | Managed | View credential details (read-only); revoke credential; for Tyk-managed custom credentials: rotate credential | Create, edit, and assign credentials via Admin Portal or Portal API | Introduced in v1.12; rotate and revoke support added in v1.17. Admin-only creation. Rotating a Tyk-managed custom credential replaces the custom key with a Tyk-generated token. Non-Tyk-managed credentials (external key-value pairs) do not support rotation. See [Custom Credentials](/tyk-developer-portal/import-custom-credentials). |

**Unsupported Authentication Types:** HMAC, OpenID Connect (OIDC), and Keyless are completely unsupported by the Portal.
### Self-Serve vs Managed

**Self-Serve:** The API Consumer creates, rotates, and revokes their own credentials from the Live Portal without Admin involvement.

**Managed:** The Portal Admin imports or pre-configures credentials. Consumers have view-only access, or can revoke but not create or rotate credentials.

## OAuth 2.0 and DCR

All OAuth 2.0 authentication in the Developer Portal uses [Dynamic Client Registration (DCR)](/tyk-stack/tyk-developer-portal/enterprise-developer-portal/api-access/dynamic-client-registration). The Tyk internal OAuth provider (`oauth` auth type) is not supported.

### Supported Identity Providers

| Identity Provider | Support | Notes |
|---|---|---|
| Keycloak | Fully supported | N/A |
| Okta | Fully supported | Requires the `OktaAppType` field (`web`, `native`, `browser`, or `service`). For Client Credentials flow, set the allowed response type to `token` only. |
| Gluu | Fully supported | N/A |
| Curity | Supported via Other type | Curity requires authentication on its DCR endpoint by default. Workaround: disable DCR endpoint authentication in Curity (no-authentication mode). |
| Any RFC-compliant IdP | Supported via Other type | Select **Other** in the DCR template settings. Full compatibility is not guaranteed. |

## Combination Rules

All APIs in an API Product must use a compatible authentication method. The Portal enforces this at checkout: a credential issued for one auth type cannot authenticate requests to an API expecting a different type.

**Single-type Products:** Any supported auth type can be used alone in a Product.

**Multi-type Products:** mTLS must be present in any Product that combines auth types. The only supported multi-type combinations are:

- mTLS + API Key
- mTLS + JWT

API Key and JWT cannot be combined without mTLS.

**Auth types excluded from all multi-type combinations:**

- Basic Authentication (`basicAuth`)
- Custom Authentication middleware (`customAuth`)

These auth types must be used alone and cannot be combined with any other auth type.

**JWT requirement:** JWT APIs must have DCR enabled. The Portal rejects JWT APIs that do not have a DCR template configured at checkout.

**DCR cross-IdP configurations:** A Product can technically reference DCR templates from multiple Identity Providers, but this configuration does not work as intended. At checkout, the consumer selects one IdP. Tokens from that IdP are rejected by any API in the Product configured to expect tokens from a different IdP. Use a single DCR template per Product.
