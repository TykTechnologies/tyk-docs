When you add APIs to an API Product, the Developer Portal enforces that all APIs in the product use an authentication method. This page summarises which authentication types the Portal supports, what consumers and admins can do with each, and which types can be combined.

<Note>
This page covers authentication as managed by the Developer Portal. For Tyk Gateway-level authentication configuration, see [API Authentication](/api-management/authentication).
</Note>

## Credential Ownership: Self-Serve vs Managed

Before reviewing the table, it helps to understand the two credential ownership modes the Portal uses:

**Self-Serve:** The API Consumer creates, rotates, and revokes their own credentials from the Live Portal without Admin involvement.

**Managed:** The Portal Admin imports or pre-configures credentials. Consumers have view-only access, or can revoke but not create or rotate credentials.

The **Mode** column in the table below indicates which model applies to each authentication type.

## Supported Authentication Types

| Auth Type | Supported | Mode | Consumer Actions | Admin Actions |
|---|---|---|---|---|
| [**API Token / API Key**](/api-management/authentication/bearer-token) | Yes | Self-Serve | Request access, rotate credential, revoke credential | View, edit, and revoke credentials |
| [**OAuth 2.0 (via DCR)**](/api-management/authentication/oauth-2) | Yes | Self-Serve | Request access (generates Client ID and Secret), rotate secret, revoke credential | Configure DCR template, view credentials, revoke credentials |
| [**JWT**](/basic-config-and-security/security/authentication-authorization/json-web-tokens) | Yes | Self-Serve | Request access, rotate credential, revoke credential | Configure DCR template, view credentials, revoke credentials |
| [**mTLS (Self-Serve)**](/api-management/authentication/certificate-auth) | Yes | Self-Serve | Upload client certificate during access request, revoke access | View credentials, revoke access |
| [**mTLS (Managed)**](/api-management/authentication/certificate-auth) | Yes | Managed | Request access using organization certificates, view credentials, revoke access | Pre-configure and manage certificates per organization in the Admin Portal |
| [**Auth Token + mTLS**](/api-management/authentication/certificate-auth) | Yes | Self-Serve or Managed | Request access; issued credential is automatically bound to the consumer's certificate | Enable certificate-token binding on the Product |
| [**Basic Authentication**](/api-management/authentication/basic-authentication) | Yes | Managed | View credential details (read-only); revoke credential | Create, edit, and assign credentials via Admin Portal or Portal API |
| [**Custom Authentication**](/api-management/authentication/custom-auth) | Yes | Managed | View credential details (read-only); revoke credential | Create, edit, and assign credentials via Admin Portal or Portal API |
| [**Custom Credentials (Feature)**](/tyk-developer-portal/import-custom-credentials) | Yes | Managed | View credential details (read-only); revoke credential; for Tyk-managed custom credentials: rotate credential | Create, edit, and assign credentials via Admin Portal or Portal API |

**Unsupported authentication types:** [HMAC](/basic-config-and-security/security/authentication-authorization/hmac-signatures), [OpenID Connect (OIDC)](/api-management/single-sign-on-oidc), and [Keyless](/basic-config-and-security/security/authentication-authorization/open-keyless) are not supported by the Portal.

## OAuth 2.0 and DCR

All OAuth 2.0 authentication in the Developer Portal uses [Dynamic Client Registration (DCR)](/tyk-stack/tyk-developer-portal/enterprise-developer-portal/api-access/dynamic-client-registration). The Tyk internal OAuth provider (`oauth` auth type) is not supported.

### Supported Identity Providers

| Identity Provider | Support | Notes |
|---|---|---|
| Keycloak | Fully supported | N/A |
| Okta | Fully supported | Requires the `OktaAppType` field (`web`, `native`, `browser`, or `service`). For Client Credentials flow, set the allowed response type to `token` only. |
| Gluu | Fully supported | N/A |
| Curity | Supported via Other type | Curity requires authentication on its DCR endpoint by default. Workaround: disable DCR endpoint authentication in Curity (no-authentication mode). |
| Any RFC-compliant IdP | Supported via Other type | Select **Other** in the DCR template settings. Full compatibility is not guaranteed. |

## Combination Rules

All APIs in an API Product must share a compatible authentication method. The Portal issues a single credential that must work across every API in the product.

### Supported combinations

Any supported auth type can be used alone. For multi-type products, only these combinations are valid:

- [mTLS](/api-management/authentication/certificate-auth) + [API Token / API Key](/api-management/authentication/bearer-token)
- [mTLS](/api-management/authentication/certificate-auth) + [JWT](/basic-config-and-security/security/authentication-authorization/json-web-tokens)

[API Key](/api-management/authentication/bearer-token) and [JWT](/basic-config-and-security/security/authentication-authorization/json-web-tokens) cannot be combined without mTLS.

### Types that must be used alone

[Basic Authentication](/api-management/authentication/basic-authentication) and [Custom Authentication](/api-management/authentication/custom-auth) cannot be combined with any other auth type.
